# Canvas Web Agent Evaluation Dataset

This repository contains the data required to initalize a Canvas environment in a state that facilitates the execution of tasks by web agents. 

## Manifest

|File(s) | Description |
|-|-|
|`tasks.json` `tasks.yaml` | **Task Definitions:** The dataset tasks are defined in [tasks.yaml](./tasks.yaml)/[tasks.json](tasks.json). They include the templates for each task, the answer keys used by the evaluation script to validate task success from network logs, as well as 10 instances for each task.  
|`cascon-2026.yaml`|**Course Data Used to Generate Environment for Tasks:** [cascon-2026.yaml](cascon-2026.yaml) contains realistic LLM generated course data that is loaded into Canvas to facilitate the execution of the tasks defined in `tasks.yaml`. |
|`cascon-2026.sql`| a database dump of Canvas after the generated course data has been loaded. Used to reload the state of the environment between runs.|
|`odobot_tasks.json` | The evaluation task instances in OdoBot format.|
|`webvoyager_tasks.jsonl`| The evaluation task instances in WebVoyager format.|
|`agent_e_tasks.json`| The evaluation task instances in Agent E format.|

>While tasks.json/tasks.yaml include `Information Seeking` tasks only `Side-effect` tasks are fully supported.

>The database snapshot has only been tested on the version of Canvas used to develop the environment setup scripts. A repository with that version of Canvas will be made available in the future. 



# CASCON 2026 Evaluation Dataset

From the 460 side-effect task instances defined in `tasks.yaml` a random selection of 46 task instances for model construction and then 46 task instances for evaluation was generated using the following script.

`python3 task_selection.py --input ./tasks.json --format both --random-sample 1 --training-csv training_instances.csv`

>Task are only generated in OdoBot and WebVoyager format, as the [Agent-E repository](https://github.com/EmergenceAI/Agent-E) includes its own tools for converting WebVoyager tasks into Agent-E tasks.

[LogUI](https://github.com/logui-framework) upon which OdoX (our firefox browser extension for recording trajectories) was built, stores trajectories as sessions inside 'flights' (trajectories). Flight data is stored in LogUI server's Mongo database. The recordings we want to use for model construction are subsequently scraped into elasticsearch with a script. `cascon-flight-list.json' lists the flights that were scraped for the model created for the Cascon paper, the flight ids should align with those listed under [OdoBot Model Construction Task Instances](#odobot-model-construction-task-instances).
